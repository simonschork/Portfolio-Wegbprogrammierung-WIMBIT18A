Webanwendungen und System-Architektur

Webanwendungen sind Anwendungsprogramme, die auf einer Client-Server-Architektur basieren. Vergleicht man Webanwendungen mit nativen Anwendungen stößt man abgesehen von der Entwicklung sofort auf einen markanten Vorteil, nämlich die Distribution der Software. Bei einer nativen Anwendung muss eine Kopie einer Installationsdatei verteilt werden. Eine Webanwendung hingegen ist unter den nötigen Voraussetzungen, wie Internetverbindung und Laufzeitumgebung in Form eines Browsers, jederzeit, und geräteunabhängig aufrufbar. Sie muss nicht installiert werden. Stattdessen wird auf das Server-Client-Prinzip als Architektur zurückgegriffen. Der Client bietet hauptsächlich die Benutzeroberfläche während der Server die Ausführung der Rechenoperationen und Datenverarbeitungsprozesse übernimmt. Dem Entwickler steht größtenteils frei, in welchem Ausmaß er Teile der Software im Client oder auf dem Server ausführen lässt.
Am Beispiel einer Internetseite in der einfachsten Form sendet der Internetbrowser einen HTTP-Request an den Webserver, dieser stellt daraufhin die HTML-Dokumente bereit und sendet sie zurück an den Client, der sie interpretiert und darstellt. Ist eine Webseite so vielseitig, dass sie nicht in jeder Variation als HTML-Dokument gespeichert werden kann, gibt es ein Modell, in dem die Daten auf einer Datenbank gespeichert werden. Diese ist nach Stichworten organisiert und liefert auf Abfrage eine Liste mit Links. Aus diesen erstellt der Server ein HTML-Dokument, was er anschließend an den Client sendet. Beliebte Programmiersprachen für serverseitige Programmierung sind z.B. Java, Perl und PHP. Wird Inhalt auf dem Client interaktiv dargestellt geschieht das meist unter Einbindung von Javascript-Code.

Um die vorher genannte Architektur von Webanwendungen erneut aufzugreifen, man kann die Programmierung hauptsächlich client-seitig vollziehen, dies nennt man eine Rich-/Thick-/Fat Client Architektur. Wobei der Großteil der Anwendung auf dem Client ausgeführt wird. Oft werden hierfür Laufzeitumgebungen von Drittanbietern erfordert, wie z.B. der Flash-Player, der lange Zeit sehr beliebt war. Diese müssen allerdings wieder auf dem Rechner des Nutzers installiert werden. Zu ihren Vorteilen gehört eine schnellere Verarbeitung da keine Requests an Server geschickt werden müssen und somit auch weniger Auslastung von Server und Netzwerk. Außerdem ist es einfacher Aufwendige GUIs zu entwickeln, da viele der Frameworks vielseitige APIs besitzen. Nachteil hingegen ist die erhöhte Belastung des Clients, somit auch eventuell längere Ladezeiten in Abhängigkeit des genutzten Clients und Sicherheitslücken, die durch die Nutzung der Drittanbieter-Software entstehen können.

Eine weitere Möglichkeit ist die gesamte Anwendung server-seitig zu implementieren. In der sogenannten Thin Client Architektur dient der Client nur noch als Ein- und Ausgabemedium, mit dem der Nutzer interagiert. Das Programm wird vollständig auf einem Server ausgeführt und die Ergebnisse an den Client gesendet. Vorteil ist hier du Unabhängigkeit. Der Kern des Programms kann in einer beliebigen Sprache geschrieben sein und das Ergebnis für den Client entsprechend übersetzt. Es muss jedoch klar sein, dass bei einer großen Anzahl an Anfragen der Server stark belastet wird und entsprechende Kapazitäten benötigt.

Heute mir oftmals eine Mischung aus beiden Architekturen eingesetzt. Um die Vorteile zu kombinieren und die Nachteile möglichst zu neutralisieren. Gerade bei der Erstellung interaktiver Webanwendungen ist es schneller und spart Serverkapazität, wenn man die Verarbeitung und Interpretation der Inhalte dem Client überlässt. Auf der anderen Seite ist es mit Hilfe des Servers und oft auch Datenbanken möglich, der Anfrage entsprechend, unterschiedliche HTML-Dokumente zurückzugeben. Zum Beispiel bei Suchanfragen können die Ergebnisse baukastenartig zusammengesetzt werden.Ein weiterer Vorteil ist die Plattformunabhängigkeit. Da Internetbrowser auf fast allen Plattformen vertreten sind, bieten sie eine einheitliche Schnittstelle und Laufzeitumgebung und mindern den Entwicklungsaufwand enorm. Ein gravierender Nachteil hingegen ist meiner Meinung nach der Notwendigkeit einer Internetverbindung. Diese hängt stark von Standort und weiteren Umweltfaktoren ab und ohne eine bestehende Verbindung zum Webserver ist die aktive Nutzung einer Webanwendung nicht möglich.

Javascript - Java's kleiner Bruder?

Javascript ist keinesfalls mit Java zu verwechseln, es handelt sich um zwei vollkommen verschiedene Programmiersprachen. Zur Abgrenzung ist es nützlich in drei Kategorien zu unterteilen: Programmiersprachen, Auszeichnungssprachen (Markup languages) und Skriptsprachen.
Die meisten Programmiersprachen, wie z.B. C++ werden durch einen Compiler entsprechend der festgelegten Regeln, von Quellcode, Zeile für Zeile in Maschinensprache übersetzt. Der Compiler übersetzt das vollständige Programm bevor es ausgeführt wird. Skriptsprachen wie Python oder Javascript werden von einem Interpreter interpretiert, es werden nur die für den Ablauf des Programms benötigten Abschnitte interpretiert, außerdem haben sie einfachere Struktur und schwächere Regeln. Die Interpretation ist zwar langsamer als die Übersetzung, allerdings benötigen sie weniger Voraussetzungen und können leichter in andere Software integriert werden. Skriptsprachen waren ursprünglich für die Lösung kleinerer, wiederkehrender Probleme gedacht. Sie sind modular aufgebaut und verlassen sich auf die Nutzung existierender Funktionen. Durch die schwächeren semantischen Regeln erscheinen sie einfacher und es ist meistens einfacher sie zu erlernen. Mit der Zeit wurden sie aber zu vollständigen Sprachen, die den klassischen Programmiersprachen in nichts nachstehen.

Das beliebteste Beispiel ist wahrscheinlich Python. Python wurde 2019 in einer Studie der IEEE erneut als beliebteste Programmiersprache ausgezeichnet. Python folgen Java auf Platz 2 und C auf Platz 3, Javascript ist auf Platz 6. Sie war auch für mich die erste Programmiersprache, mit der ich Kontakt hatte, es gibt deutlicher weniger Syntaxregeln zu beachten was sie für Anfänger besonders attraktiv macht. Aktuell ist sie besonders im Bereich maschinelles Lernen sehr beliebt, da es eine große Anzahl an Libaries gibt, die bereits fertige Algorithmen enthalten sie nur noch implementiert werden müssen.

Javascript wurde mit dem Fokus auf Webprogrammierung entwickelt. Sie kann vom Webbrowser interpretiert und über den <script>-Tag direkt in HTML-Code eingebunden werden. Sie bildet die Basis für dynamische Webentwicklung, also Funktionen und interaktive Elemente innerhalb der Webseite. Nur als Beispiel können Nutzereingaben in Textform oder Interaktionen mit Buttons aufgefangen und verarbeitet werden.

Alternativ zu direkter Einbindung in den HTML-Code, können die Funktionen in einer .js-Datei geschrieben und diese über <script src="Beispiel.js"></script> aufgerufen werden. Eine weitere Besonderheit von Javascript ist die Typisierung, hier wird nämlich „Dynamische Typisierung“ oder „loose typing“ verwendet. Die Variablen werden nur mit dem Schlüsselwort „var“ deklariert, welchen Datentyp die Variable annimmt und wie viel Speicherplatz reserviert wird, wird anhand des Inhalts vom Javascript-Interpreter bestimmt. DOM - Document Object Model

Das Document Object Model ist eine Repäsentation der Dokumentenstruktur. Diese wird vom Browser erstellt und im Arbeitsspeicher abgelegt. DOM ist die Schnittstelle zwischen HTML und Javascript. Alle Elemente werden zu Objekten (Knoten) innderhalb des DOMs, auf die dynamisch zugegriffen werden kann und welche gegebenenfalls manipuliert werden können.

Während der Browser den HTML-Code empfängt, verarbeitet er ihn Schritt für Schritt und legt dabei die Objektstruktur an und legt sie in Form eines Baumes ab:

Sobald dies einmal durchgeführt wurde nutzt der Browser nur noch die Objektstruktur und greift nicht mehr auf den Quellcode zu. Die zuvor genannten Knoten können verschiedenen Kategorien angehören, beispielhaft sind: Elementknoten, Attributknotenund Textknoten. Der Programmierer kann auf verschiedene Arten auf ein solches Objekt bzw. Knoten zugreifen:

  1. document.getElementById(): Greift über die ID, die der Entwickler mithilfe von <div id="..."> innerhalb des ersten Tags festlegen kann zu. Die ID eines Elements ist einzigartig und somit eindeutig.

  2. document.getElementByTagName(): Greift über den Tag auf mehrere Elemente zu. Die verschiedenen Elemente werden in einer Liste, vergleichbar mit einem Array gespeichert. Sie besitzen die lengt-Eigenschaft und der Entwickler kann über sie iterieren.

Eine beliebte Methode ist oft genutzte Referenzen in Variablen zu speicher um so den Zugriff zu erleichtern. Nach dem Speichern in einer Variable können auch beide oben genannten Möglichkeiten per Punktnotation miteinander verküpft werden. Da die Objekte als Baum strukturiert sind, lässt sich über verschiedene Methoden, auch auf Elter-, Kind- und Geschwisterknoten zugreifen.

Hat man nun erfolgreich die gewünschte Komponente erreicht gibt es eine Vielzahl von Manipuationsmöglichkeiten hier zwei Beispiele wie man Inhalt einer Seite verädnern kann:

  1. Die erste Option ist document.wirte(). Mit dieser Methode können neue Inhalte hinzugefügt, allerdings keine bestehenden verändert werden. Außerdem funktioniert dies nur während des Ladens der Seite.

  2. Mit .innerHTML() funktioniert das wesentlich einfacher und dynamischer. Damit greift man auf den HTML-Code eines vor dem Punkt definierten Knoten zu und kann diesen wie gewünscht verändern. Zusätzlich ist es möglich den Inhalt des Knoten in Variablen zu speichern und innerhalb des Javascriptcodes weiterzuverwenden.

Für ein Beispiel praktischer Anwendung schauen Sie sich die Kommentarseite dieser Webseite und den Javascriptcode der Funktion kommentarSenden() an.

Trotz des großen Nutzen den der DOM mit sich bringt gibt es auch einige Probleme:

  • Es ist äußerst Zeitaufwendig sich innerhalb der Baumstruktur von Knoten zu Knote zu bewegen.
  • Es gibt keine Möglichkeit alle Tags mit einer bestimmten Klasse auszuwählen.
  • Die verschiedenen Browser interpretieren z.B. Events innerhalb des DOM oder auch Tabulatoren und Leerzeichen unterschiedlich.

Als Lösung stehen uns verschiedene Javascript-Bibliotheken zur Verfügung, die diese Probleme behandeln, so dass der Entwickler seine Zeit effektiver Nutzen kann. Ein Beispiel hierfür ist jQuery, was im nächsten Abschnitt detaillierter erläutert wird.

jQuery - DOM-Handling

jQuery ist die mit Abstand meistverwendete Javascript-Bibliothek, laut W3C nutzt die Hälfte der Webseiten jQuery. Daher wird es auch in viele CM-Systemen (z.B. Wordpress) und Webframeworks mitgeliefert. Ursprünglich 2006 entwickelt von John Resig wird es aktuell vonm "jQuery Team" weiterentwickelt. Wesentlich Funktionen des Frameworks sind:

  • CSS-Selektoren
  • DOM-Manipulation
  • Event-Handling
  • Animationen und Effekte
  • Ajax-Funktionalitäten
  • Erweiterbarkeit durch Plug-ins

Die Nutzung von jQuery bietet viele Vorteile unter anderem:

  • Wenig Speicherverbrauch: Die Bibliothek lässt sich auf bis zu 30 KByte komprimieren.
  • Webdesignerfreundlich: Es baut auf CSS auf, was es für Nicht-Informatiker leichter macht das Framework zu nutzen.
  • Große Entwicklergemeinde: Auch durch die kostenfreie Nutzung gibt es Tausende Entwickler die sich mit der Bibliothek beschäftigen und sie so laufend verbessern.

Um jQuery in der Webseite zu verwenden, muss man die Datei jquery.js importieren, welche auf http://www.jquery.com/ frei zum Download verfügbar ist. Man kann sie dort in unkomprimierter, komprimierter oder minimierter Form herunterladen. Interessant ist hier die minimierte Form, ein Konzept, welches ich zum ersten Mal in Zusammenhang mit Webprogrammierung hörte. Sie ist etwas größer als die gepackte Datei, muss dafür aber nicht jedes mal entpackt werden. Schaut man sich eine solche Datei an wirkt sie auf den ersten Blick sehr verwirrend. Es handelt sich um genau den selben Code nur werden jegliche Form und Leerzeichen, die für den Syntax nicht essenziell sind gelöscht. Der Code wird in einem langen Block dargestellt. Das und die wegfallende Entpackung verringert die Ladezeit der Webseite. Auf dieser Seite habe ich mir das Runterladen von jQuery gespart und importiere die minimierte Datei über eine URL von Google. Diese vorgehensweise macht mich allerdings abhängig von deren Webserver und der Verfügbarkeit des Links.

Bevor ich genauer auf die Möglichkeiten von jQuery eingehe, was ist überhaupt ein Framework?

Bei einem Framework handelt es sich, wie der Name schon andeutet um einen Rahmen, durch den sowohl Client als auch der Programmierer auf bereitgestellten Code zugreifen. Es gibt somit im Wesentlichen die Anwendungsarchitektur vor und stellt Klassen und Funktionen zur Verfügung. Zur Erstellung dieser Webseite nutze ich ein Framework namens Bootstrap. Ich kann problemlos vordefinierte Klassen in meine HTML-Struktur einfügen und spare mit so einiges and Arbeit. Ebenso problemlos kann ich diese Klassen überschreiben oder neue Klassen definieren. Ein Framework zu nutzen eignet sich besonders für Anfänger, da man schneller Fortschritte sieht und nicht zu tief in die Funktionalitäten der verschiedenen Sprachen gehen muss. In anderen Fällen ist es aber auch unerlässlich, so würde ich behaupten keiner schreibt die Funktionen und Strukture die mit jQuery nutzbar werden selbst. Mittlerweile sind Frameworks für die Webentwicklung essentiell geworden. Sie unterstützen den Nutzer bei der Erstellung dynamischer Webseiten mit hierarchisch aufgebauter, zuverlässiger Navigation, bei der Behandlung verschiedener Events (Nutzer und Browser) und bei der Validierung von Eingaben.

Die Methoden von jQuery lassen sich in drei Kategorien einordnen:

  1. Transparente Methoden sind ohne Einschränkung verkettbar, Rückgabewert ist ein, der Eingabe-Kollektion entsprechendes jQuery-Objekt. Mit ihnen kann man die Elemente der Kollektion verändern, die Kollektion wird aber unverändert an die nächste Kollektion weitergegeben.

  2. Destruktive Methoden haben als Rückgabewert eine reduzierte, erweiterte oder ganz andere Kollektion. Sie sind auch ohne Einschänkung verkettbar.

  3. Terminierende Methoden geben kein jQuery-Objekt, sondern z.B. DOM-Knoten oder einen beliebigen anderen Datentypen zurück. Da keinen Kollektion zurückgegeben wird, kann keine jQuery-Methoden angewendet und diese so nicht verkettet werden.

Dies ist nur eine kleine Auswahl und einfachste Beispiele. jQuery bietet enorm viele Möglichkeiten wie Filter oder das Verändern von Tagattributen (z.B. Klassen).

Ajax - Asynchronous JavaScript and XML

Asynchrones Javascript and XML oder Ajax verspricht interaktivere, reaktionsschnellere und einfacher nutzbare Webanwendungen, die sich wie Desktopanwendungen anfühlen. Dies soll durch eine asynchrone Datenübertragung zwischen Browser und Webserver ermöglicht wernden. Das bedeutet die beiden Komponenten sind nicht aufeinander abgestimmt, man spricht von einer Absichtlichen zeitlichen Versetzung. Ajax greift das typische Problem der Webanwendungen auf: Da jede Benutzerinteraktion eine Anfrage an den Webserver darstellt ist die Stabilität und Wartezeit der Anwendung von der Verbindung und der sofortigen Antwort stark abhängig. Des Weiteren muss der Webserver bei Veränderung eine neu generierte Webseite and den Browser senden, was bei schlechter Verbindung ebenfalls träge wirken kann.

Ajax-Anwendungen hingegen fordern nur die tatsächlich benötigten Daten vom Server an, während dieser die Daten läd kann der Nutzer weiter mit der Anwendung interagieren.

Inhaltliche Gestaltung von Web-Content

Eine Webseite ist ein Medium das eine Vielzahl von Menschen erreichen und ansprechen soll, um dies zu gewährleisten sollte man sich bei dem Entwurf einer solchen Seite auf mindestens drei grundlegende Faktoren fokusieren: Content, Design und Usability. Natürlich ist der Inhalt wichtig, denn ohne diesen gibt es für kaum jemanden einen Grund die Seite überhaupt zu besuchen. Das Design hat essentielle Auswirkungen auf die Wahrnehmung der Nutzer. Vor allem wenn die Webseite für ein Unternehmen oder seriöses Vorhaben sein soll, lässt das Design schon erste Vermutungen über den Charakter des Unternehmens oder Vorhabens zu. Nutzerfreundlichkeit ist eben so unerlässlich, eine schlecht zu "surfende" Webseite kann einiges an Frustration bergen und wird unwahrscheinlich ein zweites mal besucht.

Zuerst ist es sinnvoll sich über die Gliederung und den Einstieg in die Webseite gedanken zu machen. Der Nutzer sollte nicht als erstes mit einem Werbebanner oder einem langen Textblock konfrontiert werden. Die Startseite sollte eher einen Überblick über die Inhalte der Seite verschaffen. gleichzeitig ist es sinnvoll sie möglichst simpel zu halten, da zu viel Information oder auch Animation den Nutzer "erschlagen" könnte. Es ist sinnvoll den Inhalt der Webseite pyramidenartig darzustellen, was so viel bedeutet, wie die wichtigen Inhalte vorranzustellen und den Nutzer, selbst entschieden zu lassen worüber er tiefer gehende Informationen haben möchten. Bei der weiteren Gliederung ist es wichtig eine allgemein logische Einteilung der Inhalte vorzunehmen. Diese sollte am Besten auch von Kollegen, oder bei einem privatem Projekt, Bekannten gezeigt werden, um eine zu einseitig subjektive Annahme auszuschließen.

Ebenso nützlich ist es sich von Anfang an auf ein Layout und Schreibstil festzulegen. Es macht einen unseriösen und "zusammengewürfelten" Eindruck, wenn jede Seite der Webseite unterschiedlich aussieht. Außerdem ist die Anordnung verschiedener Objekte, die auf jeder Seite identisch oder zumindest ähnlich sein sollen, so einfacher. Inkonsistenz und Redundanz sind zwei der Hauptgründe warum Nutzer sich nicht länger auf einer Webseite aufhalten möchten. Ein klares und übersichtlichs Design hingegen verstärkt die Wirkung von inhaltlich gutem Content. Auch für den Entwickler ist eine klare und übersichtliche Struktur von Vorteil. Sie erleichtert es Teile der Anwendung oder Webseite wiederzuverwenden und den Code zu warten. Die zurvor beschriebene Struktur sollte im Idealfall durch Menüs und Verlinkungen implementiert werden, dies ist eine Möglichkeit zu verhindern, dass der Nutzer mit Informationen "überhäuft" wird. Ein häufiger Fehler ist unter Anderem zu viele "eye catcher" einzubbauen, denn auch eine optische Überforderung ist möglich und sollte vermieden werden. Je einfacher eine Webseite oder Anwendung gestaltet ist desto wahrscheinlicher ist ein anhaltender und wiederkehrender Aufenthalt von Nutzern.

Als letztes möchte ich auf den Punkt User Interface zu sprechen kommen, bzw. auf das Thema Usability. Schluss endlich beschreiben die zuvor genannten Themen nichts Anderes. Der Begriff Usability ist ein sehr breit gefasster Terminus. Dennoch stieß ich bisher mehrmals auf die Definition als Teilbereich der praktischen Aktzeptanz eines Systems, dieser ist ausschlaggebend dafür wie gut Nutzer seine Funktionen einsetzen können. Sie lässt sich anhand der Erlernbarkeit, der Effizienz, der intuitiven Nutzung und der subjektiven Wahrnehmung bzw. Zufriedenheit des Nutzers bestimmen. Als ein Designkriterium gilt Visibility, also dem Nutzer den aktuellen Systemstatus anzuzeigen. Ebenso die zuvor bschriebene Auffindbarkeit verschiedener, für die Seite inhaltlich oder auch strukturell nützlicher Inhalte. Abschließend muss der Entwickler sich auch in die Situation eines eventuell unbedarften Nutzers hineinversetzen und versuchenn mögliche Fehlerquellen zu eliminieren. Interaktionen des Nutzers mit der Webseite oder -anwendung sollten entweder für jeden eindeutig ersichtlich oder ausreichend dokumentiert bzw. erklärt sein. So kann zum Beispiel beim Ausfüllen von Onlineformularen schon das Format der Eingabe zu vielen Problemen führen. Ein unterschätzter doch wichtiger Punkt ist also Empathie und die Fähigkeit sich in verschiedenste Typen von möglichen Nutzern hineinzuversetzen.
